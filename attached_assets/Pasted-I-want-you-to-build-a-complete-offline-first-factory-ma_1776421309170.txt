## Very important business rules
Implement these rules carefully:

### Sales
- Sales invoice must support customer info, date, driver, vehicle, notes.
- Invoice items must support item/material, quantity, price, total.
- Must track both pallet count and brick/tile count where relevant.
- Must calculate received amount, remaining debt, discount, invoice total.

### Customer payments
- A separate screen/form to register customer payments.
- Payment must reduce that customer's debt.
- Customer account statement must show invoices, payments, debt movement.

### Old debt
- Must support opening balances for customers and suppliers.

### Purchasing
- Purchasing workflow must be similar to sales, but reversed financially.
- Purchase invoice must support supplier, items/materials, quantity, price, total, paid amount, remaining debt.

### USD purchasing
- Some purchases happen in USD.
- The factory base accounting is in IQD.
- When purchase/payment is in USD, the system must use the daily exchange rate.
- It must convert USD impact into IQD for cashbox/accounting effect.

### Exchange rate
- Create a module for daily USD exchange rate.
- Store date + rate.
- Make purchase/payment calculations use the selected daily rate when needed.
- The current USD exchange rate must always be visible in the top area/header of the system.
- The user must be able to manually edit/change the USD exchange rate whenever needed.
- The user must also be able to set or update the exchange rate together with its date.
- The top header display should show the latest active rate clearly, using Kurdish UI text and English numerals.

### Materials
- Create a materials/items module.
- Each material should support purchase price, optional profit margin, and sale price if needed.
- Even if the factory mainly sells bricks, this module is important for purchase and cost tracking.

### Employees / HR
- Each employee must have a unique code.
- Support name, role, phone(s), address, guarantor/reference info, and attachment/file if possible.
- Do not hard delete employees when they leave.
- Use active/inactive status.

### Payroll
- Payroll must support employee work records and payment records.
- Show total due, paid amount, remaining amount.
- Support monthly report, yearly report, individual employee report, and payroll list.

### Expenses and other income
- Must have categorized expense records.
- Must have categorized non-sales income records.
- These records must affect final financial reporting.

### Cashbox / general financial dashboard
- Create a clear central financial dashboard.
- The dashboard must combine effects from:
  - sales receipts
  - customer debt collection
  - purchases
  - supplier payments
  - payroll
  - expenses
  - other income
  - shareholder payouts
- The result should show current cash-on-hand clearly.

### Profit / loss
- Must support period-based profit calculation.
- Profit/loss should consider sales, expenses, payroll, and other financial movement.
- Create a monthly financial closing report.

### Shareholders / shares
- This is very important.
- The factory has a share-based ownership model.
- Do not hardcode shares in a rigid way.
- Build a real Shareholders module.
- I must be able to:
  - add a new share/shareholder
  - edit existing shares/shareholders
  - record shareholder withdrawals/payouts
  - calculate share-based profit/loss distribution
- Make the design flexible, because the share setup may need editing.

### Alerts
- Add alert/report areas for overdue debts or important pending balances.

## Data integrity rules
This is critical.
- Avoid dangerous hard delete patterns in business records.
- Prefer soft delete or inactive status where appropriate.
- Preserve accounting history.
- Do not break relationships between invoices, payments, payroll, expenses, and reports.
